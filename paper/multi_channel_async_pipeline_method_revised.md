# 多通道异步推测流水线调度

本文仅讨论解码阶段。请求进入推测解码前，端侧草稿模型和服务器侧目标模型均已完成 Prompt 预填充，并持有各自的 KV cache。服务器部署 \(M\) 个参数一致、相互独立的目标模型副本，每个副本及其本地任务队列构成一条验证通道 \(G_g\)。

本文的请求内异步验证受 DSI 中 speculation parallelism 思想的启发，即利用多个目标模型实例提前执行同一请求的未来验证任务，并沿目标模型确认的有效路径依次推进验证结果。不同于 DSI 主要面向单请求推测过程，本文将该思想扩展到端边多请求服务场景，以草稿片段作为验证和调度单元，并重点解决多个请求共同竞争有限目标验证通道时的任务选择、通道分配和结果有序生效问题。

传统同步端边推测解码需要等待当前草稿完成验证后才能进入下一轮，批验证还会引入组批等待和请求间同步。本文以草稿片段为调度单元：端侧沿当前预测路径连续产生片段，服务器将同一请求的不同片段分配到多条验证通道提前执行，并同时交错处理其他请求。方法由请求内异步推进和请求间多通道调度组成。对于每个请求，所有验证任务都从同一已确认前缀出发，并沿不同深度的未确认片段路径提前执行，因此可统一表示为一棵共享前缀的分支验证树；多个请求的分支验证树共同构成动态验证森林。

## 1. 请求内异步推进

### 1.1 草稿片段与依赖关系

对于请求 \(r\)，端侧基于当前已确认前缀 \(P_r\) 连续生成草稿片段

\[
S_{r,0},S_{r,1},\ldots,S_{r,j}.
\]

后继片段建立在前序片段给出的乐观路径上：

\[
S_{r,j}\leftarrow P_r\Vert S_{r,0}\Vert\cdots\Vert S_{r,j-1}.
\]

每个片段携带请求标识、路径版本、片段序号及其直接前驱。服务器据此建立片段之间的依赖关系，并判断验证结果是否仍属于当前有效路径。

对于单个请求，当前乐观预测路径可表示为

\[
P_r\rightarrow S_{r,0}\rightarrow S_{r,1}\rightarrow\cdots .
\]

不同验证任务均以同一已确认前缀 \(P_r\) 为根，但可以在不同通道上验证到不同深度。例如，验证任务 \(V_{r,0}\)、\(V_{r,1}\) 和 \(V_{r,2}\) 分别覆盖以 \(S_{r,0}\)、\(S_{r,1}\) 和 \(S_{r,2}\) 为末端的前缀路径。它们共享祖先前缀，却拥有各自的执行状态和验证结果，因此本文将该请求的并行验证状态抽象为一棵**共享前缀的分支验证树**。

这是一种逻辑状态抽象，而不是要求所有片段在一次目标模型前向中组成物理 token tree。当请求当前只沿一条乐观路径继续生成时，树在 token 依赖上可退化为链；多个提前执行的验证任务则对应这条共享前缀路径上不同深度的验证分支。任一前序片段被拒绝后，所有路径中包含该错误后缀的后代验证任务都会失效。

同一请求的不同片段可以被分配到不同验证通道。因此，后继片段不必等待前序片段完成验证即可提前执行，但其结果只有在所有祖先片段均被确认后才能生效。

### 1.2 前沿验证任务

服务器为每个请求维护一个确认前沿。紧邻确认前沿、完成后能够直接推进请求输出的任务称为**前沿验证任务**；位于其后的任务称为**后继验证任务**。

任一时刻，每个请求只有一个前沿验证任务。后继验证任务可以提前执行，但其结果在被提升为前沿验证任务之前仅作为条件结果保留。当前沿验证任务通过时，其直接后继被提升为新的前沿验证任务；若该后继已经完成，则立即处理其结果。

例如，任务完成顺序可能为

\[
V_{r,2},\quad V_{r,0},\quad V_{r,1},
\]

但请求仍按照

\[
V_{r,0}\rightarrow V_{r,1}\rightarrow V_{r,2}
\]

推进确认前沿。

### 1.3 拒绝与路径失效

若前沿验证任务在片段内部发生拒绝，服务器提交拒绝位置之前已接受的 token，并提交目标模型在拒绝位置给出的修正 token。随后更新请求的有效路径版本，使所有依赖错误后缀的后继任务及其结果失效。

若片段全部通过，服务器接受该片段并处理目标模型给出的 bonus token。若下一片段已存在，设其首 token 为 \(x\)，bonus token 为 \(b\)：

- 当 \(b=x\) 时，下一片段仍属于有效路径，\(x\) 视为已经确认，保留其余验证结果；
- 当 \(b\neq x\) 时，服务器提交 \(b\)，并使下一片段及其后代失效。

尚未开始的失效任务直接从队列删除；已经执行的任务不进行抢占，其迟到结果通过路径版本检查丢弃。若修正 token 或 bonus token 为 EOS，请求结束并清理全部后继任务。

由此，验证任务可以乱序执行，但确认前沿之前的输出始终与目标模型的贪婪自回归解码结果一致。

## 2. 请求间多通道调度

服务器同时维护多个请求的分支验证树：

\[
\mathcal F=\{\mathcal T_1,\mathcal T_2,\ldots,\mathcal T_R\}.
\]

请求内分支验证树描述共享前缀、任务先后关系及后代失效范围；全局调度器从整片验证森林中选择任务，并将其分配到有限的验证通道。该部分是本文异步流水调度的重点。

### 2.1 前沿优先的任务选择

对于任务 \(V_{r,j}\)，定义：

- \(c_{r,j}=0\)：该任务为前沿验证任务；
- \(c_{r,j}=1\)：该任务为后继验证任务；
- \(d_{r,j}\)：任务距离当前确认前沿的依赖层数；
- \(a_{r,j}\)：任务进入候选队列的时间。

任务优先级表示为

\[
\pi_{r,j}=(c_{r,j},d_{r,j},a_{r,j}),
\]

并按字典序从小到大选择。该规则依次保证：

1. 前沿验证任务优先于后继验证任务；
2. 后继任务中，距离确认前沿较近者优先；
3. 同类任务按照到达顺序先到先服务。

新到达的前沿任务可以排在尚未开始的后继任务之前，但不抢占已经执行的目标模型前向。当前沿任务不足以占满全部通道时，剩余通道才用于执行后继任务，以提前准备后续确认结果。

严格的前沿优先使系统在高负载时优先推进当前可提交输出；后继任务的提前执行是对剩余验证能力的机会式利用，而不是对前沿任务的替代。

### 2.2 验证通道分配

选定任务后，调度器将其分配到预计最早可用的验证通道。设 \(A_g\) 为通道 \(G_g\) 的预计可用时间，则

\[
g^*=\arg\min_g A_g.
\]

由于当前设置中各目标 GPU 同构、目标模型副本一致，同一任务在不同通道上的验证时间相同，因此选择最早可用通道即可最小化其预计完成时间。同一请求不固定绑定到某条通道；其前沿任务和后继任务可以随通道状态动态分配。

### 2.3 调度状态更新

任务完成后，调度器更新对应请求的分支验证树和全局候选队列：

- 后继任务完成时，仅记录完成状态和验证结果；
- 前沿任务通过时，将其直接后继提升为新的前沿任务；
- 新前沿任务若已经完成，则立即继续处理；
- 前沿任务拒绝或发生 bonus token 分歧时，删除无效后代；
- 新片段到达时，将其加入对应请求树并重新参与全局排序。

因此，请求内前沿推进和请求间资源调度相互解耦：分支验证树维护单个请求的共享前缀、有效路径和失效后代，验证森林调度决定有限目标验证资源在多个请求之间的使用次序。

## 3. 树形 KV 管理

KV cache 管理参考 DSI 对多目标服务器的处理方式，并借鉴 SpecInfer 的树形 KV 管理。每条验证通道维护本地目标模型 KV cache；对于单个请求，所有验证分支共享同一已确认前缀，后续片段的 KV 状态按照分支验证树中的祖先关系组织。这样，共享前缀只需作为共同根状态维护，路径拒绝后则可按照树中的后代关系统一标记并回收无效 KV。

服务器同时处理多个请求，因此多棵请求级分支验证树共同组成 KV 森林。不同请求分别维护自己的根前缀和有效分支，森林管理器统一维护各请求树节点与验证通道本地 KV 状态之间的对应关系。具体的 KV 页面布局、同步方式和底层树形注意力实现沿用现有树形 KV 管理机制，不作为本文调度算法的重点展开。

## 4. 与异构感知动态起草的接口

本方法将已经生成并到达服务器的草稿片段视为调度输入，不决定片段长度、生成时机和连续起草深度。上述决策由异构感知动态起草方法完成。

动态起草模块根据端侧设备能力和当前流水线状态产生后继片段；异步流水调度模块负责在多个请求之间选择验证任务、分配验证通道并处理验证结果。二者通过片段及其依赖元数据连接，因此动态起草策略可以独立变化，而不改变请求内分支验证树的前沿推进和请求间验证森林调度的基本语义。

综上，本方法以共享前缀的分支验证树组织单个请求的提前验证状态，通过前沿推进保证结果按序生效，并以多个请求组成的动态验证森林进行前沿优先调度，从而在多个请求之间分配有限目标验证资源，避免跨请求组批等待和同步屏障。异构感知动态起草进一步决定如何持续产生适量后继片段，以形成完整的端边异步推测流水线。
