# 多通道异步推测流水线调度

本文仅关注大模型推理的 steady-state autoregressive decoding 阶段，不改变 Prompt 预填充流程。请求进入异步推测解码阶段前，草稿模型和目标模型已经分别完成当前已确认前缀的 KV cache 初始化；预填充时延及其 KV 构建过程不纳入本文的调度与加速模型。本文不建模 prompt prefill computation、prefill batch 干扰、prompt 传输、TTFT、hidden state 或 prompt 通信；主要指标为 decode-only TPOT/TBT、请求完成时延、有效吞吐和相对基线加速比。

针对端边协同推测解码中服务器侧验证受单通道串行执行和同步等待限制的问题，本文提出一种多通道异步推测流水线调度方法。服务器部署 \(M\) 个参数一致且相互独立的目标模型副本，每个副本运行于一张 GPU，并与其本地任务队列和 KV cache 共同构成一条验证通道 \(G_g\)。不同于将请求固定绑定到某个目标模型副本，本文以草稿片段为调度基本单元，使同一请求的连续未来片段能够被分派至不同通道进行乐观验证，同时允许每条通道交错处理来自不同请求的片段。由此，端侧连续起草、服务器异步调度与多 GPU 并行验证共同形成持续推进的推测流水线。

该方法包含两个相互配合的机制。首先，**Dependency-aware Multi-channel Asynchronous Verification Scheduling** 以草稿片段为任务粒度，完成片段依赖记录、跨通道分配、乱序验证、按序提交和依赖失效。其次，**Request-level Versioned KV Support** 维护请求的已确认前缀版本及其在不同验证通道上的目标模型 KV 副本，使任意通道能够按需获得片段验证所需的已确认上下文。前者决定片段的执行位置和逻辑提交顺序，后者为片段跨通道执行提供上下文状态支持。

## 1. Dependency-aware Multi-channel Asynchronous Verification Scheduling：依赖感知的多通道异步验证调度

对于请求 \(r\)，端侧草稿模型基于当前已确认前缀 \(P_r^{v_0}\) 连续生成草稿片段

\[
S_{r,0},S_{r,1},\ldots,S_{r,j},
\]

其中 \(v_0\) 表示该轮乐观起草开始时记录的已确认前缀版本。由于端侧无需等待前一片段完成验证，第 \(j\) 个片段建立在当前乐观预测路径上，即

\[
S_{r,j}\leftarrow
P_r^{v_0}\Vert S_{r,0}\Vert\cdots\Vert S_{r,j-1}.
\]

端侧上传片段时仅传输当前片段的 token，并附带请求标识、已确认前缀版本、片段序号和前序依赖片段标识。可将其依赖信息表示为

\[
D_{r,j}=\bigl(v_0,j,d_{r,j}\bigr),
\]

其中 \(d_{r,j}\) 表示当前片段直接依赖的前序片段。该信息用于恢复片段的假设上下文，并在前序片段被拒绝时识别所有受影响的后续片段；它只描述片段之间的逻辑依赖，不要求服务器形成树形计算结构。

服务器包含验证通道 \(G_0,G_1,\ldots,G_{M-1}\)。同一请求的连续片段可以被映射到不同通道，例如

\[
S_{r,0}\rightarrow G_0,\qquad
S_{r,1}\rightarrow G_1,\qquad
S_{r,2}\rightarrow G_2.
\]

因此，多通道调度同时利用请求间并行和请求内乐观并行：不同请求的片段可以并行占用不同验证通道，同一请求中尚未提交的未来片段也可以提前进入不同通道执行。后者不要求等待前序片段先完成验证，但可能在不同通道上重复计算尚未提交的公共前序 token。

### 最新兼容提交前沿与验证块

对于片段 \(S_{r,j}\)，服务器在任务实际执行时确定与其假设路径一致的最新提交前沿。设 \(c_{r,j}\) 表示该前沿已经完整提交的最后一个片段序号，\(v_{r,j}^{c}\) 表示对应的已确认前缀版本；若尚无前序片段提交，则令 \(c_{r,j}=-1\) 且 \(v_{r,j}^{c}=v_0\)。当前任务只需验证提交前沿之后仍未确认的连续 token 块

\[
B_{r,j}^{(c)}
=
S_{r,c_{r,j}+1}\Vert\cdots\Vert S_{r,j}.
\]

验证通道从最新兼容的已确认前缀 KV \(K_{r,g}^{v_{r,j}^{c}}\) 出发，将 \(B_{r,j}^{(c)}\) 作为一个连续 token 块输入目标模型，并在一次前向传播中完成前序未确认 token 的临时上下文恢复和当前片段验证。已经提交且与当前片段假设路径一致的前序 token 由 prefix KV 直接复用，不再重复计算。

本文沿用推测解码理论分析中的并行验证时延近似。目标模型能够在一次前向传播中并行验证多个 token，并且对于本文调度过程中形成的验证块，其墙钟时间近似于目标模型的一次自回归解码时间：

\[
\widehat{V}_{r,j,g}
=
\mathcal{V}_g\!\left(\left|B_{r,j}^{(c)}\right|\right)
\approx T_T,
\]

其中 \(T_T=T_{\mathrm{decode}}(1)\)。该近似用于刻画并行验证对解码关键路径的影响，并不表示多 token 验证与单 token 生成具有相同的算术工作量。

### 通道选择

设片段 \(S_{r,j}\) 在时刻 \(t_{r,j}\) 到达服务器，通道 \(G_g\) 当前的预计可用时间为 \(A_g\)。片段在该通道上的预计完成时间定义为

\[
\widehat{T}_{r,j,g}
=
\max(A_g,t_{r,j})
+
\widehat{V}_{r,j,g}.
\]

其中，\(\max(A_g,t_{r,j})-t_{r,j}\) 表示片段在该通道上的预计排队等待时间。调度器选择预计完成时间最小的通道：

\[
g^*=\arg\min_g\widehat{T}_{r,j,g}.
\]

当多个片段竞争验证资源时，各请求中紧邻当前提交前沿、完成后能够直接推进输出的片段优先；更深的未来片段利用其余验证能力提前执行。该规则避免深层乐观任务长期阻塞当前可提交结果，同时保留请求内验证重叠的机会。

### 乱序完成、按序提交与依赖失效

不同验证通道的排队状态和执行时刻不同，同一请求的多个片段可能乱序完成。服务器为每个请求维护结果缓冲区和提交前沿。提前完成的片段首先进入结果缓冲区，只有在以下条件同时满足时才能推进提交前沿：

1. 该片段记录的已确认前缀版本仍属于当前有效路径；
2. 所有前序依赖片段均已完整接受；
3. 该片段的假设上下文与当前实际提交路径一致。

若片段被全部接受，服务器按序提交该片段的草稿 token，并继续检查缓冲区中是否存在已经完成且依赖条件满足的后继片段。本文不依赖目标模型额外生成的 bonus token；片段全部接受时仅提交片段自身的草稿 token，下一位置继续由后续片段验证。

若片段在某一位置发生拒绝，服务器提交拒绝位置之前已被目标模型接受的草稿 token，以及目标模型在该位置产生的修正 token，并据此形成新的已确认前缀版本。所有直接或间接依赖被拒绝后缀的后续状态统一失效：尚未开始的任务从队列中取消，已经完成但尚未提交的结果从结果缓冲区中丢弃，正在执行的过期任务即使返回也不再参与提交。随后，端侧基于新的已确认前缀开始新的乐观起草轮次。

该过程始终维持如下提交不变式：在任意时刻，提交前沿之前的 token 序列与目标模型在相同输入上的贪婪自回归解码结果完全一致。多条验证通道可以乱序执行未来片段，但用户可见输出只沿目标模型确认的有效路径按序推进。

## 2. Request-level Versioned KV Support：请求级版本化 KV 支持

草稿模型与目标模型相互独立，端侧草稿模型产生的 KV 状态不能直接作为服务器目标模型的 KV cache 使用。本文不改变预填充流程：请求进入解码阶段前，草稿模型已经持有当前已确认前缀对应的 draft-model KV，且至少一个目标验证通道已经持有相同前缀对应的 target-model KV。此后的异步推测解码均以这些已初始化状态为起点。

对于请求 \(r\)，服务器维护唯一的逻辑已确认前缀 \(P_r^v\)，其中 \(v\) 随提交过程单调递增。各验证通道按需维护该版本的目标模型 KV 副本

\[
K_{r,g}^v,
\]

其中 \(g\) 表示验证通道。逻辑上，\(P_r^v\) 是请求当前唯一有效的已确认上下文；物理上，同一版本的目标模型 KV 可以同时存在于多个通道中。

当片段被分配至某一验证通道时，若该通道尚未持有请求的最新已确认 prefix KV，则系统在后台 KV 流中从其他通道复制最新副本，或在已有旧副本的基础上补充新增的已确认 KV。本文将这两种情况统一视为“按需同步最新已确认 prefix KV”，不进一步讨论 KV 页布局、复制路径和缓存替换等工程细节。

尚未确认的前序草稿不要求在通道之间迁移 KV。目标通道获得最新兼容的 prefix KV 后，直接将提交前沿之后的前序依赖 token 与当前片段组成验证块 \(B_{r,j}^{(c)}\)，并通过一次目标模型前向完成临时 suffix KV 构建和片段验证。由此，跨通道同步的对象仅为已经确认的 prefix KV，而未确认 suffix 的上下文恢复被融合在正常验证过程中。

每个验证通道使用主计算流执行目标模型验证，并使用后台 KV 流复制或更新已确认 prefix KV。当片段进入通道队列后，系统提前发起所需的 KV 同步，并使其与主计算流中已有的验证任务重叠；片段真正开始验证前，通过事件依赖保证所需 prefix KV 已经就绪。本文假设服务器内部目标模型副本间的 KV 传输时间显著小于一次目标模型验证时间，且其大部分能够被后台执行隐藏，因此重叠后暴露在关键路径上的 KV 开销近似为

\[
T_{\mathrm{KV}}^{\mathrm{exposed}}
=
\max\!\left(0,T_{\mathrm{KV}}-T_{\mathrm{overlap}}\right)
\approx 0.
\]

因此，KV 同步不作为独立的串行阶段计入片段预计完成时间。请求级版本化 KV 支持只负责保证任意验证通道能够获得正确的已确认上下文，而片段依赖、结果提交和失效处理均由第一部分的异步调度协议统一完成。

综上，依赖感知的多通道异步验证调度将同一请求的未来片段跨目标模型副本进行乐观执行，并通过结果缓冲、按序提交和依赖失效维持目标模型贪婪解码语义；请求级版本化 KV 支持则通过已确认 prefix KV 的按需同步，为片段跨通道执行提供上下文基础。二者共同形成端侧持续起草、服务器多通道并行验证和结果有序提交的异步推测流水线。
