# 实验思路：真实模型语义行为 + 虚拟设备理论时延 + 仿真器调度流程

## 1. 实验定义

本文采用**模型在线参与的解析时延仿真**。

对于每个请求，仿真器维护真实 token 状态，并在每轮动态起草中调用真实 draft model 生成候选 token，再调用真实 target model 按照推测解码规则计算 acceptance。因此，acceptance 并非人为设定或预先 trace 给定，而是在仿真过程中由真实模型实时产生。

与此同时，模型 forward 的实际 wall-clock time 不作为系统时延；草稿生成、目标验证和通信开销均由解析公式计算。该设计将推测解码的语义行为与硬件运行时噪声解耦，使我们能够在可控的异构设备算力模型下评估调度策略。

核心划分为：

```text
真实模型：负责语义行为
虚拟设备：负责理论时延
仿真器：负责调度流程
```

---

## 2. 多草稿模型与多虚拟设备

本文实验采用**多草稿模型、多虚拟设备**设定。

真实加载多个草稿模型，用它们在仿真过程中生成 draft tokens，并由 target model 在线验证得到对应的 acceptance。不同草稿模型具有不同的速度、计算成本和 acceptance 表现，因此能够反映草稿模型异构性。

设备数量仍然由仿真器虚拟出来。也就是说，实验不需要真实部署大量端侧设备，而是在仿真器中构造多个虚拟设备，并为每个设备设置理论算力、带宽、可部署模型集合等参数。

因此：

```text
真实加载的对象：draft model 类型、target model
虚拟出来的对象：设备数量、设备算力、设备带宽、设备调度状态
```

如果有 3 种草稿模型和 100 个虚拟设备，只需要真实运行这 3 种草稿模型，而不需要在 100 个真实设备上分别部署模型。

---

## 3. 仿真时延公式

本文只借鉴已有工作的**仿真时延建模思想**，即不用一次实验中的 wall-clock time 直接作为系统时延，而是通过解析公式估计计算与通信开销。

草稿生成时延可建模为：

```math
T_{draft}(i,m,\gamma)
=
\eta_{i,m}
+
\frac{\gamma}{r_{i,m}}
```

其中：

- `i` 表示虚拟设备；
- `m` 表示草稿模型；
- `γ` 表示本轮动态起草 token 数；
- `r_{i,m}` 表示虚拟设备 `i` 运行草稿模型 `m` 的等效 token 生成速率；
- `η_{i,m}` 表示固定启动或调度开销。

也可以写成计算工作量 / 设备算力形式：

```math
T_{draft}(i,m,\gamma)
=
\eta_{i,m}
+
\frac{\gamma \cdot w_m}{C_i}
```

其中：

- `w_m` 表示草稿模型 `m` 生成单个 token 的计算工作量；
- `C_i` 表示虚拟设备 `i` 的归一化计算能力。

目标验证时延可建模为：

```math
T_{verify}(\mathcal{B})
=
\eta_T
+
\frac{|\mathcal{B}| \cdot (\max_{r \in \mathcal{B}} \gamma_r + 1)}{r_T}
```

其中：

- `B` 表示本轮验证 batch；
- `γ_r` 表示请求 `r` 的 draft token 数；
- `r_T` 表示 target verifier 的等效验证速率；
- `η_T` 表示 target verifier 的固定开销。

通信时延可建模为：

```math
T_{comm}
=
\frac{S_{up}}{B_{up}}
+
\frac{S_{down}}{B_{down}}
```

其中：

- `S_up` 表示上传 draft tokens 或相关验证数据的数据量；
- `S_down` 表示返回验证结果或输出 token 的数据量；
- `B_up` 和 `B_down` 分别表示上行和下行带宽。

## 4. 在线仿真流程

完整流程如下：

```text
1. 加载真实 target model 和多个真实 draft model。
2. 构造多个虚拟设备，并设置每个设备的理论算力、带宽和可部署 draft model。
3. 每个请求维护真实 token 状态。
4. 每轮调度器为请求选择虚拟设备、draft model 和 draft token 数 γ。
5. 调用真实 draft model 生成 γ 个候选 token。
6. 调用真实 target model 验证候选 token，得到真实 acceptance。
7. 仿真器根据 acceptance 更新请求 token 状态。
8. 仿真器不使用真实 forward wall-clock time，而是用解析公式计算 draft、verify 和 communication latency。
9. 仿真器根据理论时延推进设备状态、队列状态和请求完成时间。
10. 输出吞吐、平均时延、首 token 时延、加速比、设备利用率等指标。
```

## 5. 与已有工作的关系

本文只借鉴已有边缘推测解码工作的**解析时延建模思想**，即通过模型计算负载、设备算力和通信带宽估计系统时延，而不是直接采用真实 wall-clock time。

本文不直接采用其完整系统模型，也不直接照搬其具体公式。本文的实验重点是：

```text
真实模型在线产生 acceptance；
虚拟设备提供理论异构时延；
仿真器评估调度策略。
```

---

## 6. 一句话总结

本文实验采用：

> 真实 draft / target 模型负责在线语义行为和 acceptance，多个虚拟设备负责理论时延建模，仿真器负责调度、排队和请求状态推进。

因此，实验不需要真实部署大量设备，但需要真实加载不同类型的草稿模型，以得到不同草稿模型对应的真实 acceptance。
