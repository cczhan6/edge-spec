# 异构感知异步推测解码：系统框架与方法设计

## 3. System Overview

如图 X 所示，本文提出一种面向异构端边环境的异步推测解码框架。系统由多个异构端侧设备和一个多通道边缘验证服务器组成。每个端侧设备独立维护自身请求及一个本地草稿模型，并沿当前推测路径生成草稿片段；边缘服务器部署 \(M\) 个参数一致的完整目标模型副本，每个副本构成一条独立验证通道。本文聚焦解码阶段，提示词预填充沿用常规处理方式。

该框架包含两个核心机制。服务器侧的**多通道异步验证**利用空闲目标模型副本提前验证后继片段，并通过乱序执行、按序提交保证最终输出正确。端侧的**异构感知动态起草**根据设备侧片段准备时间、服务器级归一化后继完成速率和近期连续接受长度，动态决定下一片段的长度以及需要维持的流水线深度。

请求进入解码阶段后，端侧首先生成启动片段并发送至服务器。随后，端侧无需等待该片段的验证结果，即可沿当前预测路径继续生成后继片段。对于请求 \(r\)，紧邻已确认前缀、验证后能够直接推进输出的任务称为**当前片段验证任务**；依赖尚未确认前序片段的任务称为**后继片段验证任务**。服务器始终优先执行当前片段任务，只有在没有等待中的当前片段任务时，才将空闲通道用于后继片段验证。

后继片段可以先于其前序片段完成验证，但不能越过前序依赖直接提交。若某个前序片段发生拒绝、bonus token 改变后续路径，或者生成 EOS，则依赖原路径的后继片段和缓存结果均失效。已经开始执行的失效任务不被抢占，其结果返回后直接丢弃。

整个在线过程形成如下闭环：

\[
\text{动态起草}
\rightarrow
\text{多通道异步验证}
\rightarrow
\text{按序提交与错误路径清理}
\rightarrow
\text{更新下一轮起草}.
\]

在任一时刻，\(M\) 条验证通道最多并行执行 \(M\) 个验证任务。每个端侧草稿模型同一时刻只生成一个片段，但同一请求可以同时存在正在生成、传输、等待验证、正在验证或等待提交的多个片段。每条验证通道不采用微批处理，同一时刻只执行一个任务，任务完成后可以切换到任意请求。

当当前片段任务占满全部通道时，系统自然退化为请求间并行验证；当存在剩余通道时，系统进一步形成同一请求不同片段之间的提前验证。动态起草不追求无限增加在途任务，而是利用片段长度和目标流水线深度减少端侧生成、通信与服务器验证之间的可避免空泡。

与 DSI 对并行推测状态的抽象处理一致，本文在逻辑上借鉴 SpecInfer 基于共享前缀组织推测 KV cache 的基本思想。由于每个请求仅沿一条推测路径生成，其请求内结构可视为树结构的单分支退化形式，不同请求之间相互独立。本文不研究目标模型副本之间的 KV cache 同步、迁移和物理存储机制；在系统模型和时延计算中，假设各验证通道能够直接获得任务所需的已确认前缀 KV 状态，并忽略相关管理开销。所有对比方法均采用相同假设。

## 4. Multi-Channel Asynchronous Verification

多通道异步验证遵循“全局排队、当前片段优先、空闲通道即时派发、乱序执行、按序提交”的流程。其目的不是为每条通道求解复杂的任务映射，而是在不延迟可直接推进输出的当前片段任务的前提下，利用剩余验证能力提前处理后继片段。

将请求 \(r\) 当前已确认的输出前缀记为

\[
\mathbf{x}^{(r)}_{1:t_r},
\]

其中，\(t_r\) 是当前已确认的 token 数。请求 \(r\) 的第 \(j\) 个草稿片段记为

\[
\tilde{\mathbf{x}}_{r,j}
=
\left(
\tilde{x}_{r,j,1},
\ldots,
\tilde{x}_{r,j,\gamma_{r,j}}
\right),
\]

其中，\(\gamma_{r,j}\) 是该片段的草稿长度，对应验证任务记为 \(J_{r,j}\)。

令 \(k_r\) 表示请求 \(r\) 当前最早仍含有未确认 token 的片段编号。因此，\(J_{r,k_r}\) 是当前片段任务，而 \(j>k_r\) 的任务是后继片段任务。任务优先级定义为

\[
\pi(J_{r,j})
=
\left(
\mathbb I[j>k_r],
j-k_r,
t_{r,j}^{\mathrm{arr}}
\right),
\]

其中，\(t_{r,j}^{\mathrm{arr}}\) 表示任务到达服务器的时刻。该优先级首先保证当前片段任务严格优先，其次在后继任务中优先选择距离当前确认前沿更近的片段，最后按照到达顺序打破平局。浅层后继任务依赖更少的未确认片段，能够更早成为可提交结果，也具有更低的路径失效风险；因此，该排序直接来自请求内依赖关系，不属于额外的经验调度规则。

令 \(\mathcal Q_{\mathrm{cur}}(t)\) 和 \(\mathcal Q_{\mathrm{suc}}(t)\) 分别表示时刻 \(t\) 的当前片段等待队列和后继片段等待队列。当任一目标模型副本空闲时，服务器选择

\[
J^*(t)
=
\begin{cases}
\displaystyle
\arg\min_{J\in\mathcal Q_{\mathrm{cur}}(t)}
\pi(J),
&
\mathcal Q_{\mathrm{cur}}(t)\neq\varnothing,
\\[8pt]
\displaystyle
\arg\min_{J\in\mathcal Q_{\mathrm{suc}}(t)}
\pi(J),
&
\mathcal Q_{\mathrm{cur}}(t)=\varnothing.
\end{cases}
\]

选中的任务被派发到任意空闲通道。由于各目标模型副本同构，不再为通道选择引入额外优化目标。已经开始的目标模型前向不被抢占，因此后来到达的当前片段任务最多等待正在运行任务的剩余执行时间。

后继片段是在尚未确认的前序 token 上生成的，不能脱离这些前序 token 独立验证。任务 \(J_{r,j}\) 的连续验证输入记为 \(\tilde{\mathbf{x}}_{r,j}^{\mathrm{ver}}\)，它包含从当前已确认前缀之后开始、沿当前推测路径直到片段 \(j\) 末尾的全部未确认 token，并满足

\[
\left|
\tilde{\mathbf{x}}_{r,j}^{\mathrm{ver}}
\right|
\le
L_{\max}^{\mathrm{ver}}.
\]

连续验证输入之间可以包含重叠前缀，但任务 \(J_{r,j}\) 只保存和提交属于片段 \(j\) 的逐位置验证结果，包括各位置的目标 token、首次不一致位置、修正 token 和 bonus token。这样，当某个片段前缀已经由前序 bonus token 确认时，可以裁剪该前缀并继续复用剩余验证结果，而不必重新验证整个片段。

任务完成后，结果首先进入请求级结果缓冲区。仅当片段 \(j\) 之前的所有依赖均已确认，系统才按照标准推测解码规则提交该结果：

1. 依次提交从片段起点开始与目标模型一致的草稿 token；
2. 在首次不一致位置提交目标模型修正 token；
3. 若片段全部通过，则提交目标模型给出的 bonus token；
4. 若 bonus token 与下一片段首 token 一致，则该 token 直接成为已确认前缀，并从下一片段的未确认部分中裁剪；
5. 若拒绝、bonus token 不一致或 EOS 改变后续路径，则清除依赖原路径的后继片段和缓存结果。

由于每次提交都严格采用目标模型在当前已确认前缀下给出的贪婪 token，按片段顺序归纳可知，本文方法的最终输出与目标模型自回归贪婪解码一致。异步执行只改变任务开始和完成的时间，不改变 token 的提交顺序与目标模型决策。

## 5. Heterogeneity-Aware Dynamic Drafting

异构感知动态起草只保留两个相互配合的控制量：**片段长度**和**流水线提前深度**。片段长度控制调整单个草稿任务的粒度，使新片段尽量在服务器消耗现有后继任务之前到达；提前深度控制决定需要预先维持多少个尚未完成验证的后继片段，用于吸收离散长度、设备速度、通信状态和验证服务波动造成的剩余节拍失配。

两者作用不同。长度控制首先避免单个片段准备过慢直接形成验证空泡；若候选长度能够按服务器服务节拍及时到达，则目标深度通常接近一层。只有当离散长度、接受质量或连续验证长度约束使端侧无法精确匹配服务节拍时，深度控制才通过提前铺设多个片段提供短期缓冲。因此，本文不将目标深度解释为长期供需平衡保证，而将其视为服务器突发空闲能力和短期速率波动下所需的流水线提前量。

### 5.1 按时到达优先的片段长度

设离散草稿长度集合为

\[
\Gamma
=
\left\{
\gamma^{(1)},\gamma^{(2)},\ldots,\gamma^{(K)}
\right\},
\]

并包含一个较短的启动长度 \(\gamma_{\mathrm{start}}\)。对于请求 \(r\)，长度为 \(\gamma\) 的片段从开始生成到到达服务器并具备入队条件的预计时间记为

\[
T_r^{\mathrm{ready}}(\gamma).
\]

该时间由端侧草稿速度和当前通信条件共同决定，因此能够直接反映不同设备、草稿模型和网络状态之间的异构性。其估计由离线测量初始化，并使用实际片段完成时间在线平滑更新。

仅根据准备时间选择长度可能偏向目标模型难以接受的过长片段。为避免引入每请求、每长度的接受率表或未经统一校准的 softmax 置信度，本文只维护请求 \(r\) 的近期平均连续接受长度 \(\bar a_r\)。对于最终位于有效输出路径上的片段 \(j\)，令 \(a_{r,j}\) 表示从片段起点开始连续被目标模型接受的草稿 token 数，则

\[
\bar a_r
\leftarrow
\beta_a\bar a_r
+
(1-\beta_a)a_{r,j}.
\]

冷启动时，\(\bar a_r\) 使用草稿—目标模型对的离线均值；若无离线统计，则使用启动长度。因前序路径失效而被丢弃的后继结果不用于更新该统计。由于一个全部通过的短片段只能表明其全部 token 被接受，而不能观测目标模型是否还会继续接受更多 token，\(\bar a_r\) 是近期可观测连续接受长度的保守统计，而不是对真实可接受长度的精确预测。

草稿长度集合始终包含单 token 候选，即

\[
1\in\Gamma.
\]

单 token 片段不是单独的运行模式，而是动态长度控制的合法最小粒度。当端侧供给较慢、近期接受情况较差，或者连续验证输入空间有限时，系统可以自然退化到 \(\gamma=1\)。

根据保守接受长度统计定义当前长度上界

\[
\gamma_r^{\mathrm{cap}}
=
\min\left(
\gamma_{\max},
\max\left(
1,
\left\lfloor\bar a_r\right\rfloor+1
\right)
\right).
\]

其中，\(\lfloor\bar a_r\rfloor+1\) 表示在近期已经观测到的连续接受范围上最多向上探索一个 token，同时通过下界 1 保证单 token 片段始终可以被考虑。下一片段的候选集合定义为

\[
\mathcal G_{r,j+1}
=
\left\{
\gamma\in\Gamma
\;\middle|\;
1\le\gamma\le\gamma_r^{\mathrm{cap}},
\quad
n_{r,j+1}^{\mathrm{unconf}}+\gamma
\le
L_{\max}^{\mathrm{ver}}
\right\},
\]

其中，\(n_{r,j+1}^{\mathrm{unconf}}\) 表示验证下一片段时需要一同输入目标模型的既有未确认 token 数。在 \(1\in\Gamma\) 的条件下，候选集合只有在

\[
n_{r,j+1}^{\mathrm{unconf}}+1
>
L_{\max}^{\mathrm{ver}}
\]

时才为空，即连一个新的草稿 token 都无法加入连续验证输入。此时端侧暂停继续起草，并在已确认前缀推进后重新决策。

为避免请求级服务率样本过少，服务器维护一个共享的**归一化后继完成速率** \(\mu^{\mathrm{suc}}\)。令 \(C^{\mathrm{suc}}(t-W,t)\) 表示窗口内完成的后继验证任务数，\(N_{\mathrm{act}}^{\mathrm{suc}}(\tau)\) 表示时刻 \(\tau\) 至少拥有一个未完成后继任务的活跃请求数，则原始观测为

\[
\hat{\mu}^{\mathrm{suc}}(t)
=
\frac{
C^{\mathrm{suc}}(t-W,t)
}{
\displaystyle
\int_{t-W}^{t}
N_{\mathrm{act}}^{\mathrm{suc}}(\tau)\,d\tau
}.
\]

该定义以活跃请求时间为分母，表示一个活跃请求近期平均获得的后继验证服务速率。它利用服务器级样本提高稳定性，同时避免将多通道的总完成速率完整分配给每个请求。实现中使用指数滑动平均更新 \(\mu^{\mathrm{suc}}\)；窗口内没有活跃后继请求时保持上一估计。请求尚无稳定服务器观测时，继续使用启动长度。

服务器对一个活跃请求的平均后继服务间隔近似为

\[
\frac{1}{\mu^{\mathrm{suc}}}.
\]

绝对误差匹配会把“提前到达”和“延迟到达”视为等价，但延迟到达会直接造成验证空泡，而提前到达只会产生有限等待。因此，本文首先构造能够按时到达的候选集合

\[
\mathcal G_{r,j+1}^{\mathrm{on}}
=
\left\{
\gamma\in\mathcal G_{r,j+1}
\;\middle|\;
T_r^{\mathrm{ready}}(\gamma)
\le
\frac{1}{\mu^{\mathrm{suc}}}
\right\}.
\]

若 \(\mathcal G_{r,j+1}^{\mathrm{on}}\neq\varnothing\)，选择其中准备时间最接近服务间隔但不晚于该间隔的片段：

\[
\gamma_{r,j+1}
=
\arg\max_{\gamma\in\mathcal G_{r,j+1}^{\mathrm{on}}}
T_r^{\mathrm{ready}}(\gamma).
\]

该选择在避免服务器等待的前提下，尽量使用较大的任务粒度，减少过多细碎片段。若所有候选片段都无法按时到达，则选择准备时间最短的候选：

\[
\gamma_{r,j+1}
=
\arg\min_{\gamma\in\mathcal G_{r,j+1}}
T_r^{\mathrm{ready}}(\gamma).
\]

因此，长度控制遵循如下单侧原则：

\[
\boxed{
\text{优先选择能够按时到达的最大可用片段；
若端侧无法跟上，则选择最快到达的片段。}
}
\]

该规则直接面向流水线空泡，而不再将提前等待和延迟到达视为相同代价。

### 5.2 速率驱动的流水线提前深度

片段长度确定后，端侧进一步判断是否继续沿当前预测路径生成更深的片段。令

\[
d_r
=
\left|
\mathcal I_r^{\mathrm{unfinished}}
\right|
\]

表示请求 \(r\) 当前尚未完成目标模型验证的后继片段数量。其中，\(\mathcal I_r^{\mathrm{unfinished}}\) 包含正在生成、传输、等待验证和正在验证的后继片段，不包含已经验证完成但等待前序提交的缓存结果，也不包含已经失效的片段。

在准备一个新片段的时间内，一个活跃请求预计能够获得的后继验证服务量为

\[
\mu^{\mathrm{suc}}
T_r^{\mathrm{ready}}(\gamma_{r,j+1}).
\]

据此定义请求 \(r\) 的目标提前深度

\[
d_r^*
=
\max\left(
1,
\left\lceil
\mu^{\mathrm{suc}}
T_r^{\mathrm{ready}}(\gamma_{r,j+1})
\right\rceil
\right).
\]

至少维持一个直接后继片段，可以保留最基本的端侧起草与服务器验证重叠，并持续暴露后继任务以观测服务器的剩余服务能力。该下界不会为后继任务预留通道，当前片段仍具有严格优先级。

端侧是否继续起草由

\[
b_{r,j}
=
\mathbb I
\left[
d_r<d_r^*
\;\land\;
\mathcal G_{r,j+1}\neq\varnothing
\right]
\]

决定。当当前深度低于目标深度且下一片段满足连续验证长度约束时，端侧立即继续生成；否则暂停，并在片段到达、任务完成、已确认前缀推进或路径失效后重新计算。

目标深度的含义是**短期流水线提前量**。例如，当服务器在一个片段准备周期内可能处理三个后继任务时，端侧希望预先维持约三个未完成后继片段，以吸收通道突然空闲或服务速率短时升高造成的供给缺口。但该提前量不能突破端侧草稿模型的长期生成速率上限。若最短候选仍满足

\[
T_r^{\mathrm{ready}}(\gamma)
>
\frac{1}{\mu^{\mathrm{suc}}},
\qquad
\forall\gamma\in\mathcal G_{r,j+1},
\]

则服务器的长期消费速度高于该设备的草稿供给速度。增加提前深度只能利用已有草稿吸收短期突发能力，无法永久消除验证空泡。本文因此不声称目标深度能够在任意异构条件下实现长期速率平衡。

当长度控制能够实现

\[
T_r^{\mathrm{ready}}(\gamma_{r,j+1})
\approx
\frac{1}{\mu^{\mathrm{suc}}},
\]

通常有 \(d_r^*\approx1\)。深度大于一主要出现在候选长度离散、最短片段仍准备较慢、接受质量限制过长片段、连续验证输入受限，或服务器短期服务速率发生变化的情形。因此，长度控制负责优先消除单个任务的节拍失配，深度控制仅补偿剩余失配，而不是与长度控制重复优化同一目标。

当前深度采用片段数量进行轻量计数，不对生成、传输、排队和验证状态分别赋予连续权重。不同长度和不同剩余阶段的片段均被近似为一个单位，因此 \(d_r^*\) 不是精确的多服务器队列预测。本文通过限制候选长度跨度并在实验中报告实际空泡和深度收益，评估这一段级近似是否足够。

当拒绝、bonus token 不一致或 EOS 导致当前预测路径失效时，端侧直接删除依赖旧路径的本地后继片段，并从最新已确认前缀使用启动长度重新起草。回滚仅作为异步执行的正确性流程，不再维护回滚率、保守模式、试探任务或恢复状态机。深层路径的累计失效风险不再参与在线控制，而通过以下实验指标显式衡量：

\[
\mathrm{Waste}_{\mathrm{ver}}
=
\frac{
\text{最终被丢弃任务消耗的验证时间}
}{
\text{全部验证任务消耗的时间}
}.
\]

上述控制仍包含两个有意保留的轻量近似：其一，服务器级归一化速率表示活跃请求平均获得的机会服务速率，而不是单个请求未来完成时间的精确预测；其二，当前深度按尚未完成的片段数量计数，而不区分片段长度及其所处阶段。本文在方法中保持这些近似以避免引入复杂排队预测器，并在实验章节单独验证其适用范围，包括验证输入长度与目标模型时延的关系、目标深度分布、不同深度片段的有效率、验证空泡和无效验证时间。

算法 2 汇总了动态起草流程。

```text
Algorithm 2  异构感知动态起草

初始化：
  1. 使用启动长度 gamma_start 生成首个片段；
  2. 初始化不同设备的片段准备时间和平均连续接受长度；
  3. 初始化服务器级归一化后继完成速率。

服务器状态更新：
  1. 统计窗口内完成的后继验证任务数；
  2. 累计拥有未完成后继任务的活跃请求时间；
  3. 更新归一化后继完成速率。

收到有效路径上的验证结果时：
  1. 统计该片段的连续接受 token 数；
  2. 更新该请求的平均连续接受长度；
  3. 丢弃因前序路径失效而产生的统计样本。

每次需要作出起草决策时：
  1. 在包含单 token 候选的离散集合中，根据保守接受长度上界和连续验证上限构造候选集合；
  2. 若尚无稳定的服务器服务率观测，则使用启动长度；
  3. 否则优先选择能够在平均服务间隔内到达的最大候选；
  4. 若所有候选均无法按时到达，则选择准备时间最短的候选；
  5. 根据片段准备时间与归一化后继完成速率计算目标提前深度；
  6. 统计正在生成、传输、等待验证和正在验证的后继片段数；
  7. 若当前深度低于目标深度且候选集合非空，则继续生成；
     否则暂停并等待新的状态事件。

路径失效时：
  1. 删除依赖旧路径的本地后继片段；
  2. 服务器丢弃对应的等待任务和缓存结果；
  3. 从最新已确认前缀使用启动长度重新起草。
```

简化后的动态起草只需要维护设备相关的片段准备时间、每请求一个平均连续接受长度、服务器级一个归一化后继完成速率，以及每请求一个未完成后继片段计数。其核心可以概括为

\[
\boxed{
\text{按时到达优先决定片段长度}
+
\text{速率—时延乘积决定短期流水线提前量}
}.
\]
